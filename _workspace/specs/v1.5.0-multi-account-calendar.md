# v1.5.0 — True multi-account Google Calendar support

**Status:** DRAFT — open questions resolved (2026-06-17). Awaiting final
sign-off before Phase 1. No code written.

**Decisions locked:**
- **Q-A → in-app OAuth.** "Connect account" runs the OAuth flow on a background
  thread and posts the result back over the bridge. No Terminal. This folds the
  old Phase 3 into the core build.
- **Q-B → add `userinfo.email` + `openid` scope.** Reliable per-account email
  labels. One-time re-consent for the existing account on upgrade (migration
  note must warn).
- **Q-C → email-named token files** (`tokens/<email>.json`).
- **Q-D → soft cap of 5 accounts** with a UI note.
**Origin:** Settings UX review (Q1). User wants to connect/disconnect *separate
Google accounts*, not just pick more calendars under one login.
**Related:** [[v1.4.0-user-settings-json]] (sidecar pattern this reuses for the
account registry).

---

## Problem

Today CalFlow authenticates as exactly **one** Google identity:

- `secrets/token.json` — a single OAuth token.
- `secrets/credentials.json` — the OAuth client (shared, fine to keep shared).
- `data/config.json` (`{"calendars": [id, …]}`) — a flat list of calendar IDs,
  all fetched through that one token.

The current `data/calendars.json` watches four calendars
(`…@group.calendar.google.com`, `mr.hahn@gmail.com`, `cio@vanillabreeze.com`,
`david@breezm.com`) — but only because they're all **shared into** the one
logged-in account. A user with a personal Gmail and a separate work Workspace
account that are *not* cross-shared cannot watch both. There is no way to add a
second login.

## Goal

Watch calendars across **N independent Google accounts**, each with its own
OAuth token, manageable from the Settings UI without Terminal:

- Connect additional Google accounts (separate consent per account).
- Disconnect an account (revoke locally; its calendars stop being watched).
- Per-account calendar picker (which of *that account's* calendars to watch).
- Every existing single-account install migrates transparently — zero user
  action, identical behaviour after upgrade.

## Non-goals

- Cross-account dedup *configuration* (we dedup automatically; see below — but
  no user-facing knob).
- Non-Google calendar providers (iCloud, Exchange). Out of scope.
- Per-account scopes/permissions divergence — every account gets the same
  read-only scope set.
- Simultaneous OAuth flows (connect one account at a time).

## Current architecture (what must change)

| Layer | Today (single) | Consumers |
|---|---|---|
| Token | `secrets/token.json` | `calendar_client.build_service()`, `_has_oauth_token()`, `_google_account_label()` |
| Client secret | `secrets/credentials.json` | `build_service()` (shared — unchanged) |
| Watched set | `data/config.json` → `{"calendars":[id,…]}` (+ legacy `calendars.json`) | `get_selected_calendars()` |
| Service build | `build_service()` → one service | 8 call sites in `cli/main.py`, `cli/onboarding.py` |
| Fetch loop | `for cal_id in get_selected_calendars(): get_upcoming_events(service, cal_id)` | run cycle, status, missed, popover-feed |
| Aggregate | `next_event_across_calendars(service, [ids])` | status dashboard |
| Settings JSON | `calendar.{google_account, google_status, calendars_watched}` — scalar | `print_settings_json()` → settings.html Calendar section |

The single hardest constraint: **every fetch must know which account's service to
use for which calendar.** A bare `calendar_id` is no longer enough — the same
shared calendar can appear under two accounts. The unit of work becomes the
pair **(account, calendar_id)**.

## Target architecture

### Storage

```
secrets/
  credentials.json                 # OAuth client — shared, unchanged
  tokens/
    david@breezm.com.json          # one token per connected account
    personal@gmail.com.json
data/
  accounts.json                    # registry: which accounts + which calendars
```

`data/accounts.json` (gitignored — `data/` already is):

```json
{
  "schema_version": 1,
  "accounts": [
    {
      "email": "david@breezm.com",
      "token_file": "tokens/david@breezm.com.json",
      "calendars": [
        "david@breezm.com",
        "cio@vanillabreeze.com",
        "3nqcrbmg6ksh4kcm9imqbtgca0@group.calendar.google.com"
      ],
      "connected_at": "2026-06-17T00:00:00Z"
    },
    {
      "email": "personal@gmail.com",
      "token_file": "tokens/personal@gmail.com.json",
      "calendars": ["personal@gmail.com"]
    }
  ]
}
```

Token filenames are the sanitized email (`/`, `:` → `_`; emails are otherwise
filesystem-safe). The registry is the source of truth for *which* token file and
*which* calendars per account; the token files hold only OAuth material.

### Scopes

```python
GOOGLE_SCOPES = [
    "https://www.googleapis.com/auth/calendar.readonly",
    "https://www.googleapis.com/auth/userinfo.email",  # NEW (Q-B)
    "openid",                                           # NEW (Q-B)
]
```

After consent, read the granted email from the `id_token` (or the
`oauth2/v3/userinfo` endpoint) — authoritative, no more token-JSON scraping.
Migration must warn that the **existing account re-consents once** because the
scope set widened (Google invalidates a token whose scopes no longer match).

### New module: `infra/calendar/accounts.py`

```python
load_accounts() -> list[Account]          # reads accounts.json, []-safe
save_accounts(accounts)                    # atomic write
build_services() -> list[tuple[Account, service]]   # one authed service each
watched_pairs() -> list[tuple[Account, str]]        # (account, calendar_id)
add_account(email, creds)                  # write token + registry entry
remove_account(email)                      # delete token + registry entry
set_calendars(email, calendar_ids)         # update one account's watched set
list_available_calendars(account) -> [...] # that account's calendarList()
```

### In-app OAuth (Q-A) — `account-add` bridge op

The menubar is single-threaded pyobjc; `flow.run_local_server(port=0)` blocks.
The `account-add` op therefore:

1. Spawns the OAuth flow on a **daemon background thread** (`InstalledAppFlow` +
   `run_local_server(port=0, open_browser=True)`).
2. Main thread returns immediately; JS shows a "waiting for browser…" sheet.
3. On success the worker reads the account email from the `id_token`, writes the
   token file + registry entry, then resolves the pending bridge promise (via
   the `src_wv`-threaded `_resolve`, per the v1.3.5 bridge-routing rule).
4. **Timeout / cancel:** a Cancel button rejects the promise and the worker is
   abandoned after a 120 s cap; the local server socket is closed so a stale
   browser tab can't complete a half-finished grant.
5. Reject on duplicate email (account already connected) and on cap-of-5.

`build_service()` in `calendar_client.py` stays (builds *one* service from a
given token path) and becomes the primitive `build_services()` calls per
account. No behaviour change to the low-level fetch functions
(`get_upcoming_events`, `get_recent_events`) — they already take an explicit
`service` + `calendar_id`.

### Fetch loop rewrite

Every `for cal_id in get_selected_calendars(): fn(service, cal_id)` becomes:

```python
for account, service in build_services():
    for cal_id in account.calendars:
        for ev in get_upcoming_events(service, cal_id):
            ev["account"] = account.email   # tag for dedup + display
            ...
```

8 call sites in `cli/main.py` + 1 in `cli/onboarding.py`. `next_event_across_calendars`
gains a multi-service variant `next_event_across_accounts(pairs)`.

### Dedup

A calendar shared into two connected accounts yields the **same Google event
`id`** from both. Dedup the merged stream by `(calendar_id, event_id)`, keeping
the first occurrence. This is automatic and unconfigurable (non-goal above).
Lock it with a regression test: two accounts, one shared calendar, one event →
fires once.

### Settings JSON shape change

`calendar` block goes from scalar to a list. settings.html Calendar section
becomes a **custom renderer** (same mechanism as the Aliases editor —
`customRenderer:"calendar"`), not the current flat `rows`:

```json
"calendar": {
  "accounts": [
    { "email": "david@breezm.com", "status": "ok",
      "calendars": [
        { "id": "david@breezm.com", "summary": "David", "watched": true },
        { "id": "cio@vanillabreeze.com", "summary": "CIO", "watched": true }
      ] }
  ]
}
```

New bridge ops in `cli/menubar.py` (mirroring the `apply-targets` pattern):

| op | args | effect |
|---|---|---|
| `account-add` | — | runs OAuth, writes token + registry, returns new account |
| `account-remove` | `{email}` | deletes token + registry entry |
| `account-list-calendars` | `{email}` | `calendarList()` for that account |
| `account-set-calendars` | `{email, calendars:[id]}` | updates watched set |

Each restarts the daemon on success (same as `apply-targets`).

## Migration (transparent, one-time)

On first run after upgrade, if `data/accounts.json` is absent **and** legacy
`secrets/token.json` exists:

1. Probe the token's email (read `id_token` / userinfo if the email scope is
   present; else fall back to `calendarList` primary's `id`, which is the
   account email for the primary calendar).
2. Move `secrets/token.json` → `secrets/tokens/<email>.json`.
3. Build one account entry; populate `calendars` from existing
   `data/config.json` / `data/calendars.json`.
4. Write `data/accounts.json`. Leave the legacy files in place (read-only
   fallback) for one release, then remove in v1.6.

A user who never touches Settings sees no difference: same calendars, same one
account, now expressed in the new registry.

> **Scope-widening wrinkle (Q-B fallout):** the legacy `token.json` was granted
> for `calendar.readonly` only. It keeps working for *fetching* (that scope is
> unchanged), so calendars never go dark. But it has no email claim, so the
> migrated card shows the scraped/`calendarList`-primary email until the user's
> next natural re-auth. Migration therefore does **not** force an immediate
> re-consent — it migrates silently and the wider scope is acquired lazily on
> the next token refresh failure or when the user reconnects. A one-line note in
> the Settings card ("reconnect to enable email display") covers the gap. This
> keeps upgrade friction at zero while still moving to the new scope set for all
> *newly* connected accounts.

## Phasing (2 PRs)

**Phase 1 — data model + CLI + in-app-OAuth backend (no Settings UI yet).**
`accounts.py`, migration, widened scopes, `build_services()`, fetch-loop
rewrite, dedup, the `account-add` background-thread OAuth machinery, and Terminal
commands `account add` / `account remove` / `account list`. Existing installs
migrate and behave identically (modulo the one-time re-consent for the wider
scope). Fully shippable on its own.

**Phase 2 — Settings UI.**
Calendar section custom renderer: account cards, Disconnect button, per-account
calendar checklist, in-app Connect (the four bridge ops wired to Phase-1
backend), waiting/cancel sheet for the OAuth round-trip.

> Old "Phase 3" is gone — in-app OAuth (Q-A) and the email scope (Q-B) are now
> part of Phases 1–2 per the locked decisions.

  preserved, behaviour identical.

  other account untouched.
