# Missed-events recovery pane

**Status:** accepted (memo only — implementation deferred to menubar GUI)
**First accepted:** v1.1.18
**See also:** `docs/menubar_readiness.md` §7.4

## Problem

CalFlow's daemon trigger window is roughly 10½ minutes wide:

```
[event_time − ALERT_OFFSET − EARLY_TOLERANCE,
 event_time − ALERT_OFFSET + GRACE_SECONDS]
```

With defaults (`ALERT_OFFSET=300s`, `EARLY_TOLERANCE=30s`,
`GRACE_SECONDS=600s`), the window centres ~5 minutes before start.

If the laptop is asleep across the entire window, the event is **silently
skipped** — `get_upcoming_events()` is forward-only (`[now, now+FETCH_WINDOW]`)
and the trigger-window check runs BEFORE `is_done()`, so missed events never
reach state.

This is invisible to the user — they only notice "the thing didn't open."

## Proposed surface (menubar app)

A "Missed events" pane with:

- **Default look-back:** past 12 hours (configurable). Long enough to cover
  overnight sleep, short enough that the list doesn't become noise after
  a long weekend.
- **Source data:** new sibling API `get_recent_events(hours)` that mirrors
  the existing forward-only `get_upcoming_events()`.
- **Filter:** event has a `+CalFlow+` block OR Smart-mode URL, AND the
  trigger window has already closed, AND not in `state.done`.
- **Per-row action:** `[Run now]` (calls `run_event(ev)` — see
  `menubar_readiness.md §2.3`) or `[Dismiss]` (marks done in state without
  executing).
- **Background refresh:** every ~5 min while pane is open.

## Why no auto-fire on wake

- A 9 AM standup is irrelevant by 10 AM.
- Auto-firing missed events on wake violates the principle of least surprise.
- The pane gives the user agency: see what was missed, decide what's still
  useful, click through.

## CLI fallback (could ship before menubar)

```
python3 -m cli.main missed [--hours 12]
```

Lists the same data and offers an interactive `[r]un / [d]ismiss / [s]kip`
prompt per row. ~50 lines of code.

## Out of scope

- Predictive scheduling (don't fire if user is in a meeting)
- Missed-event notifications (push) — needs OS notification permission
- Recurring-event partial-miss detection

## Implementation hooks (when ready)

- `infra/calendar/calendar_client.py::get_recent_events(hours)` — mirror
  of `get_upcoming_events`, calls Google API with `timeMin=now-Nh, timeMax=now`.
- `cli/main.py::missed_command()` — CLI entry that filters + prompts.
- `state/state_manager.py` — already has `is_done` / `mark_done`; no change.
- Future menubar Swift code reads the same Python API.

## Verification

The CLI fallback's tests live in `tests/test_v2_missed.py` (TBD), with a
smoke case in `tests/test_v2_daemon_smoke.py` that asserts a missed event
is NOT auto-fired by the regular daemon cycle.
