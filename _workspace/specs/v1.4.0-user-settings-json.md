# v1.4.0 — User settings overrides via `data/user_settings.json`

**Status:** IMPLEMENTED — internal v1.4.1-dev commit
**Origin:** v1.3.14 audit follow-up. Personal preferences were leaking into
`config/settings.py` because `core/settings_writer.py` mutates the tracked
file in place on every menubar Apply.

---

## Problem

1. After any menubar Apply click, `git status` shows `M config/settings.py`
   indefinitely. Background noise that hides real diffs.
2. Personal preferences (`AUTOFILL_PROVIDER = "bitwarden"`,
   `DEFAULT_ALERT_SECONDS = 180`, etc.) live alongside project defaults in
   the same tracked file — no separation of concerns.
3. The v1.3.14 squash had to manually skip `config/settings.py` from
   staging to avoid committing PII. The next person who runs `git add -A`
   pays for the same mistake.
4. Same risk class as `config/settings.py.bak` (gitignored in v1.3.14), but
   `settings.py` itself can't simply be gitignored — defaults must stay
   tracked.

## Goal

Defaults and overrides separated into tracked defaults plus gitignored
sidecars:

- `config/settings.py` — defaults only, committed, **never written to by
  code at runtime**.
- `data/user_settings.json` — overrides, **gitignored** (`data/` is already
  in `.gitignore`), the sole write target of `core/settings_writer.py`.
- `data/user_targets.json` — user alias / `TARGETS` overrides, **gitignored**,
  the sole write target of `core/targets_writer.py`.

`git status` stays clean for users who haven't manually edited `settings.py`.

## Non-goals

- Schema validation beyond what `EDITABLE_SETTINGS` already enforces.
- Per-profile / multi-user settings (single-user app).
- Runtime hot-reload of overrides (daemon already restarts on Apply).
- Public-facing config-file format guarantees (the JSON shape is internal;
  Settings UI is the supported interface).

## Storage shape

```json
{
  "schema_version": 1,
  "overrides": {
    "AUTOFILL_PROVIDER": "bitwarden",
    "DEFAULT_ALERT_SECONDS": 180,
    "MAX_URLS": 10,
    "TITLE_URL_AUTOFILL_DEFAULT": "fill",
    "PLUS_MAX_COMMANDS": 40
  }
}
```

Only keys whitelisted by `core.settings_writer.EDITABLE_SETTINGS` may
appear. Unknown keys are dropped on load with a `[WARN]` and removed from
the JSON on next write.

## Resolution model

**Option A — populate constants at import (RECOMMEND)**

```python
# config/settings.py
from core.settings_reader import load_user_overrides

_OVR = load_user_overrides()

MAX_URLS              = _OVR.get("MAX_URLS",              5)
DEFAULT_ALERT_SECONDS = _OVR.get("DEFAULT_ALERT_SECONDS", 300)
AUTOFILL_PROVIDER     = _OVR.get("AUTOFILL_PROVIDER",     "apple")
# … one line per editable key
```

Pros: every existing `from config.settings import MAX_URLS` works unchanged.
The daemon already restarts on Apply (existing behavior), so module-load is
the override-pickup moment. Tests that monkeypatch `config.settings.X` keep
working.

Cons: import-time side effect (file read on `import config.settings`).
Acceptable — `_load_user_overrides` is one open + json.load + close.

**Option B — explicit `get()` callsites**

```python
from config.settings import get
url_cap = get("MAX_URLS")
```

Pros: clear runtime mutability.
Cons: ~50+ callsite migrations across `cli/`, `runtime/`, `core/`. No
benefit since daemon restart is mandatory anyway.

**Pick: A.** Match existing import patterns. One-shot work in `settings.py`,
no churn elsewhere.

## Writer change

`core/settings_writer.py::apply_settings` swap:

- Write target: `config/settings.py` → `data/user_settings.json`.
- Atomic-write + `.bak` pattern preserved; `.bak` now lives under `data/`
  (which is gitignored already, so no `.gitignore` change needed).
- Whitelist + type-coerce + range-check pipeline unchanged.

## Migration (one-shot, idempotent)

CLI: `python -m cli.main migrate-settings`.

1. Read `config/settings.py` in its current state.
2. Diff against the **defaults snapshot** — see open question 1.
3. For each non-default value whose key is in `EDITABLE_SETTINGS`, write to
   `data/user_settings.json` under `overrides[key]`.
4. Restore `config/settings.py` to the defaults snapshot.
5. Print a summary table: `<KEY>: <old> → <default>, override stored`.

Idempotent: if no drift detected, no-op. If `data/user_settings.json`
already exists, merge (overrides found via diff take precedence over
existing JSON values? — see open question 2).

Menubar shows a one-time "Migrate v1.3 settings?" prompt with a button
that calls this op. After successful migration, the prompt never returns.

## Affected files

- `config/settings.py` — replace each editable constant's RHS with a
  `_OVR.get(...)` call; add the `_OVR` load at top of module.
- `config/settings.defaults.py` (new, optional) — see open question 1.
- `core/settings_writer.py` — repoint write target; whitelist + atomic
  pipeline unchanged.
- `core/settings_reader.py` (new, ~50 LOC) — `load_user_overrides()`:
  open + parse + whitelist-filter + return dict. Handle missing file,
  malformed JSON, schema_version mismatch.
- `cli/main.py` — new `migrate-settings` op + JSON endpoint.
- `runtime/menubar/settings.html` — first-run migration prompt.
- `tests/test_v4_user_settings.py` (new):
  - Overrides absent → defaults applied.
  - Overrides present + valid → values applied, types coerced.
  - Overrides present + malformed JSON → defaults applied, WARN logged.
  - Unknown key in JSON → dropped + WARN + persisted out on next write.
  - schema_version mismatch (future) → migration hook called.
  - Writer roundtrip: apply → JSON written atomically → reload → match.
  - Writer atomic failure: simulated mid-write crash leaves `.bak` intact.
  - Migration: `settings.py` with drift → JSON populated, settings.py
    restored, second run is no-op.
- `tests/test_v3_settings_writer.py` — adapt expected write target.
- `docs/menubar.md` — settings-storage architecture section.

## Open questions

1. **Defaults snapshot strategy.** Two options:
   - (a) Ship `config/settings.defaults.py` alongside `settings.py` — a
     frozen copy used by the migration diff and as the import-time
     fallback for unknown keys. Explicit, but duplicates the file.
   - (b) Trust `git show HEAD:config/settings.py` at migration time —
     avoids duplication, but breaks for users on installed builds without
     `.git/` (pip / Homebrew distribution).
   - **Lean toward (a)** — explicit beats clever; the duplication is small.

2. **Override-merge precedence on re-migration.** If a user has already
   touched `data/user_settings.json` and then runs `migrate-settings`
   again, do (a) settings.py drift overwrite JSON values, or (b) JSON
   values win? — Lean toward (a): the user clicked migrate; their stated
   intent is to flush settings.py drift into the JSON.

3. **TARGETS scope.** `core/targets_writer.py` also mutates `settings.py`
   (the TARGETS dict). Symmetric problem. Bundle a `data/user_targets.json`
   split into the same release, or defer to v1.4.1? — **Resolved: bundled.**
   Keeps the architectural change coherent and removes the second dirty-file
   risk.

4. **Hot-reload.** Daemon currently restarts on Apply. A file watcher on
   `data/user_settings.json` would remove the restart requirement.
   Out of scope for 1.4.0; file as v1.5.x.

## Acceptance criteria

- [x] Fresh install: `git status` clean after first menubar Apply.
- [x] Existing v1.3.x install: `python -m cli.main migrate-settings`
      produces a populated `data/user_settings.json`, `settings.py` returns
      to defaults, daemon picks up overrides on next start.
- [x] All v3 tests pass.
- [x] All new v4 tests pass.
- [x] `docs/menubar.md` documents `data/user_settings.json` as the storage
      location for runtime preferences.

## Cost estimate

- ~250 LOC implementation + ~150 LOC tests.
- One release cycle (1.4.0).
- No DSL surface impact.

## Risks

- **Migration data loss** if the diff logic misses an edited value. Mitigation:
  always write `config/settings.py.bak` (now under `data/`) before restoring,
  and require migration to round-trip via JSON load before restoring the .py.
- **Plugin breakage** if external code reads `config.settings.X` at import
  time and expects mutation in place. CalFlow has no such plugin surface
  today; not a real risk for this codebase.
